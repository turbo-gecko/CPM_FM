| 2.13.2 | 2026-06-25 | Software Engineer | Bug fix — cancelling a transfer left the progress dialog stuck open. **Bug:** the v2.13.1 abort pushed the CAN (0x18) sequence out with `serial.Serial.flush()`, which busy-waits on `out_waiting` with no timeout. When the transmit line could not drain — the aborting remote stops asserting CTS (RTS/CTS) or sends XOFF (XON/XOFF), so flow control holds the queued bytes — `flush()` blocked forever and hung the transfer worker thread. The worker never reached `_finish_cancelled_batch`, so the `transfer_cancelled` signal was never emitted and the modal Transfer Progress Dialog (no close button; Cancel already disabled and showing "Cancelling…") stayed on screen with no way to dismiss it. Reported on host→remote, where the large queued packet data makes the stall most likely; the single shared `_abort` path means it could affect receive cancellation too. **Fix:** `_abort` now waits for the CAN bytes via a new **bounded** `_drain_tx(timeout=1.0)` that polls `out_waiting` until it reaches 0 or the timeout elapses, instead of the unbounded `flush()`; ports/stubs without `out_waiting` fall back to a single best-effort `flush()`. The abort always returns, so the worker always signals completion and the dialog is always torn down; any CAN bytes still queued when the wait elapses are transmitted by the OS in the background while the (shared transport) port stays open, so the remote still receives the abort. **Also (responsiveness):** the remaining plain `time.sleep` waits on the transfer worker thread — the FR-089 post-launch handshake delay (`_send_one_to_remote`/`_recv_one_to_host`), the FR-109 between-files settle (`_wait_for_terminal_idle`), and the FR-145 pre-upload remote listing read (`_capture_terminal_response` when called from `_fresh_remote_names`) — now poll the cancel flag via a new `_cancellable_sleep` helper and wake early, so a Cancel during any of them is observed promptly instead of after the full delay (up to ~10s). The standalone Remote-list refresh / drive-change reads keep the non-cancellable behaviour so they never observe a stale transfer-cancel flag. **Added:** OI-26. **Modified:** FR-120 (the abort's wait for the CAN to drain is bounded so a flow-control stall cannot hang the worker and freeze the dialog open), NFR-003 (the CAN-drain wait is time-bounded via `_drain_tx`, not an unbounded `flush()`), DR-041 (application/SRS version now `2.13.2`). **Code:** `terminal/xmodem.py:_abort` (`flush` → `_drain_tx`), new `terminal/xmodem.py:_drain_tx`; new `app.py:_cancellable_sleep`, with the FR-089/FR-109/FR-145 worker-thread waits (`_send_one_to_remote`, `_recv_one_to_host`, `_wait_for_terminal_idle`, and `_capture_terminal_response`/`_fresh_remote_names` via a new `cancellable` flag) routed through it. **Tests:** `tests/test_xmodem.py` (the in-memory fake serial gains an `out_waiting` property; `test_cancel_flushes_serial_in_both_directions` updated for the bounded drain; new `test_abort_does_not_hang_when_tx_cannot_drain` runs the cancel abort against a serial whose `out_waiting` never reaches 0 and asserts `send_file` returns promptly rather than hanging); `tests/test_gui_smoke.py` (new `test_cancellable_sleep_returns_immediately_when_cancelled`, `test_cancellable_sleep_completes_when_not_cancelled`, `test_wait_for_terminal_idle_returns_early_on_cancel`, `test_capture_terminal_response_cancellable_bails_early`; the `_capture_terminal_response` stubs updated for the new optional `cancellable` argument). `src/version.txt` → `2.13.2`. |
| 2.13.1 | 2026-06-24 | Software Engineer | Bug fix — flush the serial port on transfer cancellation. **Bug:** when a file transfer was cancelled, the X-Modem abort transmitted the CAN (0x18) sequence to the remote but never flushed the Transport Port, so the partially-transmitted packet still queued for transmit kept draining to the remote and incoming bytes kept arriving — the transfer appeared to keep going until the buffers emptied. The abort now flushes the Transport Port in **both** directions: it discards the queued transmit data (`reset_output_buffer`), sends the CAN and blocks until it is actually transmitted (`write` + `flush`), then discards anything the remote was still sending (`reset_input_buffer`); the ordering keeps the CAN itself from being discarded. Because the abort is the single point every cancellation path funnels through (handshake, between packets, mid-retransmit, and during receive), this covers both send and receive cancellation. **Modified:** FR-120 (the abort flushes the Transport Port transmit and receive buffers so the cancel takes effect at once), NFR-003 (CAN abort now also flushes both serial buffers, with the transmit-discard / CAN-flush / receive-discard ordering), DR-041 (application/SRS version now `2.13.1`). **Code:** `terminal/xmodem.py:_abort` (`reset_output_buffer` → `write(CAN*3)` → `flush` → `reset_input_buffer`). **Tests:** `tests/test_xmodem.py` (the in-memory fake serials gain `reset_output_buffer`/`flush`; new `test_cancel_flushes_serial_in_both_directions` asserting the cancel path resets the output buffer, flushes, and resets the input buffer). `src/version.txt` → `2.13.1`. |
