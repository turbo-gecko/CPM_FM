# CP/M File Manager — Issue Resolution Log

> Companion to the [Software Requirements Specification](cpm_fm_requirements.md) (CPM-FM-SRS).
> This is the SRS §10 Issue Resolution Log, kept in a separate file so the main specification
> stays small. All issues are closed; new resolutions are appended here (see the SRS
> "Requirement-change workflow" in `AGENTS.md`).

The ambiguities and gaps identified during the initial consolidation (v1.0) were resolved in v1.1 by
inspecting the authoritative implementation under `src/cpm_fm/`. A v1.2 requirements review (multi-agent
check against the as-built code) surfaced further code/spec discrepancies; the stakeholder decided each
direction (code-follows-spec or spec-follows-code), recorded as OI-09–OI-19 below. A v1.7.1 alignment
pass added OI-20–OI-21; later alignment passes added OI-22 (v1.7.3) and OI-23 (v2.5.2). A v2.12 bug-fix
pass added OI-24, a v2.13.1 bug-fix pass added OI-25, and a v2.13.2 bug-fix pass added OI-26. A
requirements-quality review then decomposed the monolithic NFR-003 into atomic sub-requirements,
adding OI-27, and a follow-on bloat audit decomposed five further over-bundled requirements, adding
OI-28. All issues are now closed.

| OI-25 | On a transfer cancellation (FR-120) the X-Modem abort transmitted the CAN (0x18) sequence but never flushed the Transport Port. The partially-transmitted packet still queued for transmit kept draining to the remote and incoming bytes kept arriving, so the transfer appeared to continue until the serial buffers emptied rather than stopping at once. | **Resolved (code-follows-spec).** `_abort` now flushes the Transport Port in both directions: discard the queued transmit data (`reset_output_buffer`), send the CAN and block until it is transmitted (`write` + `flush`), then discard the receive buffer (`reset_input_buffer`); the ordering keeps the CAN itself from being discarded. As the single abort path serves every cancel point (handshake, between packets, mid-retransmit, receive), this covers both send and receive cancellation. FR-120/NFR-003 reworded to state the abort flushes both buffers. | `xmodem.py:_abort`; `tests/test_xmodem.py:test_cancel_flushes_serial_in_both_directions` | FR-120, NFR-003 |
| OI-26 | The OI-25 abort used `serial.Serial.flush()` to push the CAN bytes out, which busy-waits on `out_waiting` with no timeout. When the transmit line could not drain — the aborting remote stops asserting CTS (RTS/CTS) or sends XOFF (XON/XOFF), so flow control holds the queued CAN bytes — `flush()` blocked forever, hanging the transfer worker thread. It never reached `_finish_cancelled_batch`, so `transfer_cancelled` was never emitted and the modal progress dialog (which has no close button and a now-disabled Cancel button) stayed open with no way to close it. Reported against host→remote, where the large queued packet data makes the stall most likely. | **Resolved (code-follows-spec).** `_abort` now waits for the CAN bytes via a new bounded `_drain_tx(timeout=1.0)` that polls `out_waiting` until it reaches 0 or the timeout elapses, instead of the unbounded `flush()`; ports/stubs without `out_waiting` fall back to a single best-effort `flush()`. The abort always returns, so the worker always signals completion and the dialog is always torn down; any CAN bytes still queued are sent by the OS in the background while the port stays open. FR-120/NFR-003 reworded to require the bounded drain. | `xmodem.py:_abort`, `_drain_tx`; `tests/test_xmodem.py:test_abort_does_not_hang_when_tx_cannot_drain` | FR-120, NFR-003 |
